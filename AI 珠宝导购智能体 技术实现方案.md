# **AI 珠宝导购智能体 技术实现方案**

## **1. 目标**

本文档用于将 PRD 落地为可执行的技术实现方案，重点解决以下问题：

* 大模型不是只在个别节点调用，而是每轮对话都必须参与；
* 用户新增、修改、撤回条件时，系统必须重新识别并重新推荐；
* 检索、排序、规则映射和文案生成之间的边界要明确；
* 首轮开场、追问、重新筛选、推荐输出要符合真实导购对话体验。

## **2. 总体架构**

建议采用以下架构：

1. **消息接入层**
2. **会话编排层**
3. **大模型理解层**
4. **规则映射与检索层**
5. **排序与推荐层**
6. **文案生成层**
7. **数据与日志层**

推荐技术栈：

* 后端：`Python + FastAPI`
* 数据库：`PostgreSQL` 或 `MySQL`
* 定时任务：`APScheduler` / `Celery`
* Excel 解析：`openpyxl` / `pandas`
* 模型接入：统一后端网关
* 对象存储：`OSS / COS / S3`

## **3. 核心原则**

### **3.1 每轮必经大模型**

每次收到用户消息后，都必须执行一次大模型理解流程，至少完成：

* 当前意图识别
* 条件抽取
* 条件变更判断
* 回复策略判断

若要对用户输出自然语言，则最终回复也必须由大模型生成或润色。

### **3.2 检索必须是确定性的**

大模型只负责理解和编排，不直接“脑补商品”。所有商品必须来自货盘表，价格、二维码、商品编码必须由后端结构化数据提供。

### **3.3 条件变更必须重算**

用户只要新增、修改、撤回以下条件之一，就必须重新检索：

* 预算
* 款式
* 材质
* 送礼对象
* 风格倾向
* 显贵 / 轻奢 / 性价比偏好
* 星座 / 生肖 / 生日
* 图片分析结果

### **3.4 用户可见回复必须像导购**

系统输出不能是脚本式固定问答，必须按阶段切换：

* 开场：欢迎 + 能力说明 + 轻量引导
* 追问：承接上下文后补问缺失条件
* 推荐：解释为什么推荐
* 变更后重推：明确说明“已按您刚补充的信息重新筛选”

## **4. 单轮消息处理流程**

每一轮用户输入统一走以下流程：

1. 接收消息
2. 读取会话状态
3. 调用大模型做语义解析
4. 输出结构化条件
5. 执行条件合并与冲突覆盖
6. 判断本轮动作类型
7. 如需检索，则执行规则映射、商品过滤、商品排序
8. 调用大模型生成本轮回复
9. 保存消息、状态、候选集和最终结果日志

### **4.1 动作类型**

系统每轮只允许进入以下动作之一：

* `GREETING`：开场欢迎
* `ASK_FOLLOWUP`：继续追问
* `RETRIEVE_AND_RECOMMEND`：检索并推荐
* `RERANK_AND_RECOMMEND`：已有候选集基础上重排后推荐
* `EXPLAIN_NO_RESULT`：无结果解释与兜底
* `CLARIFY_CONFLICT`：条件冲突确认
* `GENERAL_REPLY`：非推荐型自然回复

### **4.2 动作判断规则**

* 若用户是首次进入且只发寒暄语，进入 `GREETING`
* 若关键条件不足，进入 `ASK_FOLLOWUP`
* 若关键条件满足且本轮有新增或修改，进入 `RETRIEVE_AND_RECOMMEND`
* 若只是弱偏好新增，可进入 `RERANK_AND_RECOMMEND`
* 若无候选结果，进入 `EXPLAIN_NO_RESULT`
* 若本轮表达与上一轮明显冲突且置信度不足，进入 `CLARIFY_CONFLICT`

## **5. 会话状态设计**

### **5.1 会话状态结构**

建议维护一个结构化状态对象 `session_state`：

```json
{
  "budget": null,
  "budget_flexibility": null,
  "category": [],
  "material_main": [],
  "material_stone": [],
  "gift_target": null,
  "usage_scene": null,
  "style_preferences": [],
  "luxury_intent": [],
  "zodiac": null,
  "constellation": null,
  "birthday": null,
  "image_features": [],
  "excluded_preferences": [],
  "last_action": null,
  "last_retrieval_hash": null
}
```

### **5.2 状态更新规则**

* 用户最新明确表达优先覆盖旧值
* 可并存条件保留数组形式
* 否定表达写入 `excluded_preferences`
* 本轮更新后生成一个 `retrieval_hash`
* 若 `retrieval_hash` 变化，则必须重新检索

### **5.3 关键条件定义**

以下条件任一命中即可开始推荐，但命中越多结果越精准：

* 预算
* 款式
* 材质
* 送礼对象
* 风格倾向

若只有寒暄或非常泛的表达，则先追问。

若当前会话已有预算、品类或其他有效条件，且用户以“随便 / 都可以 / 不限 / 不挑 / 没要求”等表达放开剩余约束，解析层应视为开放范围委托并进入 `RETRIEVE_AND_RECOMMEND`；空上下文仅表达“随便”时仍先追问。

若用户以“啥都不想看 / 什么都不想看 / 不想看任何东西 / 先不看了”等表达拒绝继续浏览，解析层应优先进入 `ASK_FOLLOWUP`，不检索、不返回商品卡片，并提示是否换个方向或稍后再挑。

## **6. 大模型调用设计**

### **6.1 调用点**

本项目至少有三个固定模型调用点：

1. **语义理解调用**
2. **图片分析调用**
3. **推荐文案生成调用**

### **6.2 语义理解调用**

输入：

* 当前用户消息
* 最近若干轮对话
* 当前 `session_state`

输出建议为结构化 JSON：

```json
{
  "intent": "update_budget",
  "action": "RETRIEVE_AND_RECOMMEND",
  "conditions": {
    "budget": 5000,
    "category": ["项链"],
    "luxury_intent": ["轻奢", "显贵"]
  },
  "condition_changes": [
    {"field": "budget", "type": "replace"},
    {"field": "luxury_intent", "type": "append"}
  ],
  "needs_clarification": false,
  "followup_question": null
}
```

### **6.3 图片分析调用**

输入：

* 用户图片
* 图片类型提示词
* 当前会话条件

输出：

* 图片是否可用
* 自拍或参考图判断
* 风格特征
* 材质倾向
* 系统属性映射建议

### **6.4 推荐文案调用**

输入：

* 已确认的结构化条件
* 已排序候选商品
* 当前动作类型

输出：

* 面向用户的自然语言回复
* 每款商品推荐理由
* 引导语

## **7. 规则映射设计**

### **7.1 显贵 / 轻奢 / 性价比识别**

不能只做关键词硬匹配，建议采用“双层识别”：

1. 大模型先识别自然语言语义
2. 后端再映射到标准标签

标准标签示例：

* `luxury_feel`
* `light_luxury`
* `cost_effective`
* `gift_friendly`
* `mature_elegant`

### **7.2 品类映射**

建议维护同义词表：

* 手串 -> 手链
* 吊坠 -> 项链
* 耳坠 -> 耳环

### **7.3 星座 / 生肖映射**

单独维护规则表，不依赖模型自由发挥：

* 输入：星座 / 生肖 / 生日
* 输出：推荐材质、颜色、配石、风格倾向

映射后的条件参与检索和排序，不直接替代硬条件。

### **7.4 图片特征映射**

图片分析结果建议映射到：

* 系统属性
* 主材质
* 配石材质
* 风格倾向

## **8. 商品检索与排序设计**

### **8.1 检索阶段**

检索只做确定性过滤：

* 品类
* 预算
* 材质
* 是否有二维码或图片

### **8.2 预算规则**

统一以 `批发裸价` 为准：

* 默认范围：`预算 ±15%`
* 不足 3 款：放宽到 `±20%`
* 仍不足：进入兜底推荐和追问

### **8.3 排序规则**

建议采用打分制：

```text
总分 = 一级命中基础分 + 二级加权分 + 三级辅助分
```

二级加权示例：

* 适合人群命中：+30
* 显贵款命中：+20
* 系统属性命中：+15
* 卖点命中：+15
* 折扣优势：+10

三级辅助示例：

* 星座/生肖映射命中：+8
* 图片风格映射命中：+8
* 使用场景命中：+6
* 风格倾向命中：+6

### **8.4 重检索与重排边界**

以下情况必须重检索：

* 预算变更
* 款式变更
* 材质变更
* 送礼对象变更
* 星座/生肖/生日新增
* 图片分析结果新增

以下情况可先尝试重排：

* 新增“更显贵一点”
* 新增“更有性价比”
* 新增“更温柔 / 更大气”

若重排后结果变化太小，可直接进入重检索。

## **9. 输出生成设计**

### **9.1 开场输出**

示例目标：

* 欢迎用户
* 说明可帮忙推荐什么
* 引导提供预算、款式、送礼对象中的 1–2 项

### **9.2 追问输出**

要求：

* 承接已知条件
* 只问当前最缺的条件
* 不重复问已明确条件

### **9.3 推荐输出**

每次输出 3–4 款商品，包含：

* 商品名称
* 品类
* 价格参考
* 二维码或图片
* 推荐理由
* 下单引导

### **9.4 条件变更后的输出**

必须带出以下语义：

* 已识别用户新条件
* 已重新帮用户筛选
* 推荐变化原因是什么

## **12. 日志与监控**

每轮至少记录：

* 原始输入
* 大模型解析结果
* 状态变更内容
* 是否触发重检索
* 候选商品集
* 最终推荐结果
* 二维码可用性
* 异常类型

重点监控指标：

* 首轮欢迎触发率
* 条件变更后重检索触发率
* 推荐结果刷新率
* 无结果率
* 二维码失效率

## **13. 测试方案**

### **13.1 必测对话场景**

1. 用户只发“你好”
2. 用户只给预算
3. 用户给预算后回复“随便 / 都可以 / 不限”
4. 用户说“啥都不想看”
5. 用户给预算后再改预算
6. 用户先看耳饰后改看项链
7. 用户新增“我是天蝎座”
8. 用户新增“属龙”
9. 用户新增“想轻奢一点、显贵一点、但要性价比高”
10. 用户上传自拍
11. 用户上传参考图
12. 用户说“不想要珍珠了”

### **13.2 验收标准**

* 首轮寒暄不能直接机械盘问
* 条件变更后推荐结果必须刷新
* 新增星座生肖后文案必须体现识别原因
* 所有商品都必须来自真实货盘
* 二维码失效时必须降级图片

## **15. 实施结论**

本项目最关键的实现原则不是“有没有接大模型”，而是：

* **大模型要每轮都参与理解和编排**
* **后端要对真实商品做确定性过滤**
* **用户条件一变，状态、检索、排序、文案都要跟着变**

只有这样，智能体才能真正像导购，而不是像一个只会固定发问和重复推荐的脚本系统。
## 补充规则：动态优先级（2026-04）

为了避免实现跑偏，推荐逻辑补充为以下固定规则：

1. 用户明确提到的 `款式 / 品类`
优先进入强过滤，主要映射到 `系统款式`。

2. 用户明确提到的 `材质`
同样进入强过滤，优先检查 `主材质`，同时兼容检查 `配石材质`、商品名称和卖点文本，避免“用户说了水晶，但货盘写在配石或商品名里”时被漏掉。

3. 用户明确提到的 `送礼对象 / 适合人群`
进入强过滤或强排序。
例如：`男友 -> 男款`，`50岁以上女性 -> 退休妈妈款优先`。

4. 用户明确提到的 `轻奢 / 显贵 / 性价比 / 高级感 / 日常百搭`
不作为第一层硬过滤，而作为强排序信号，优先影响显贵款、系统属性、卖点和风格排序。

5. `生日 / 星座 / 生肖`
作为辅助加权信号，不应轻易把结果筛空。

一句话原则：
`用户明确说出的核心购买条件，谁被提到，谁就应进入优先检索；其中款式、材质、人群是强条件，风格与显贵诉求是强排序，生日/星座/生肖是辅助加权。`
