# **产品需求文档（PRD）：AI 珠宝导购智能体**

## **1\. 产品概述**

### **1.1 产品背景**

本公司拥有较为丰富的珠宝货盘，但外部客户在选购过程中普遍存在以下问题：

* 不清楚自己适合哪类珠宝；  
* 不熟悉商品差异，难以快速筛选；  
* 导购人工推荐效率有限，难以同时服务大量客户；  
* 现有小程序具备下单能力，但在“购买前咨询与推荐”环节缺乏智能化承接。

基于以上问题，拟依托企业微信等官方对话入口，接入多模态大模型，打造一款面向外部用户的 AI 珠宝导购智能体。该智能体负责在成交前完成需求理解、商品推荐和购买引导，最终输出商品二维码，引导用户进入公司现有小程序完成下单。

### **1.2 产品定位**

本产品不是独立商城，也不是通用聊天机器人，而是一个基于公司货盘表的\*\*“AI 导购前置系统”\*\*。

其核心职责为：

1. 与客户进行多轮对话，理解购买需求；  
2. 基于公司私有货盘表筛选真实商品；  
3. 输出推荐理由、商品图片/二维码及购买引导语；  
4. 将客户引导至公司现有小程序完成下单。

### **1.3 核心目标**

第一阶段目标如下：

* 支持客户通过文本表达预算、品类、材质偏好、送礼对象、风格倾向等需求；  
* 支持客户上传照片，进行辅助风格分析与搭配建议；  
* 基于货盘表中的真实商品信息进行推荐，禁止虚构商品；  
* 输出 3–4 款商品的推荐结果，并提供商品二维码；  
* 通过多轮追问缩小推荐范围，提升咨询效率与推荐准确性。  
* 保证大模型在每轮对话中都参与“需求理解 / 状态判断 / 输出生成”，避免智能体退化为固定问答脚本或静态推荐器；  
* 当用户补充、修改或撤回预算、款式、星座、生肖、风格等条件时，系统应重新识别并触发新一轮检索、排序与文案生成。

### **1.4 第一阶段成功标准**

建议以以下指标作为 MVP 验收参考：

* **对话完成率**：进入会话后完成至少 3 轮互动的用户占比；  
* **推荐输出率**：成功返回商品推荐结果的会话占比；  
* **二维码点击/识别率**：推荐结果中二维码被点击或识别的占比；  
* **错误推荐率**：输出不存在商品编码、失效二维码或明显不匹配商品的比例。

## **2\. 数据范围与字段定义**

### **2.1 数据来源**

系统唯一商品数据来源为**公司现有 Excel 货盘表**。系统需将 Excel 表解析为结构化数据库或可检索知识库，供智能体检索调用。

### **2.2 可用字段（固定）**

本项目以现有字段为准，不新增商品字段。可使用字段如下：

* 商品名称  
* 商品编码  
* 商品图片  
* 商品二维码  
* 产品优势卖点  
* 主材质  
* 配石材质  
* 开团价  
* 批发裸价  
* 折扣  
* 系统款式  
* 系统属性  
* 适合人群  
* 显贵款

### **2.3 字段使用说明**

* **商品名称**：用于推荐结果展示，是用户可见的核心字段。  
* **商品编码**：作为商品唯一标识，用于检索、日志记录、风控校验与异常排查。  
* **商品图片**：用于推荐结果展示；当二维码缺失、失效或不适合展示时，可作为降级输出。  
* **商品二维码**：作为优先展示字段，用于引导用户跳转至公司现有小程序进行下单或查看详情。  
* **产品优势卖点**：用于生成推荐理由、搭配建议、营销话术，是 AI 输出文案的重要依据。  
* **主材质 / 配石材质**：用于材质偏好、星座/生肖规则映射后的商品筛选。例如用户指定 K 金、珍珠、红宝石等条件时，可优先匹配。  
* **开团价 / 批发裸价 / 折扣**：用于预算筛选、价格参考和性价比推荐。  
  * **批发裸价**为用户预算匹配的核心价格字段，所有预算相关检索默认优先依据批发裸价进行筛选；  
  * **折扣**为粉丝可享优惠权益字段，可用于推荐文案中体现价格优势；  
  * **开团价**展现商品原价格 说明本商品的优势；  
* **系统款式**：定义为商品大类/品类字段，例如：手链、项链、耳饰、戒指等。  
  主要用于：  
  * 与用户明确的品类需求匹配；  
  * 推荐结果中的品类说明；  
  * 首轮推荐过滤。  
    **注意！！！**：系统款式中手链和手串为同一属性 例如用户想购买手串 系统需同时检索手串和手链 项链与吊坠也同理；  
* **系统属性**：定义为商品补充属性字段，用于表达商品的设计感、佩戴特征、风格描述或业务内部属性标签。  
  主要用于：  
  * 作为推荐理由生成时的辅助依据；  
  * 与脸型、风格、佩戴偏好做规则映射；  
  * 在候选商品排序时提供额外加权；  
  * 大模型也可参考系统属性填写推荐理由；  
* **适合人群**：用于“送礼对象”或“佩戴人群”场景下的检索与加权，例如送老婆、送妈妈、送闺蜜、自戴等。  
* **显贵款**：用于“显贵”“高级感”“预算有限但想大气”等场景下的优先推荐，大模型可进行训练。

### **2.4 字段使用原则**

由于当前项目不可新增商品字段，因此所有高级推荐能力需基于现有字段进行\*\*“规则映射”\*\*实现，而非依赖新增标签。

*示例：*

* “送女朋友”优先检索【适合人群】 注意：退休妈妈款要首先是提示词关于妈妈的 其次年龄要为50岁以上；  
* “预算 3000 内”优先筛选【批发裸价 ≤ 3000】；  
* “想要项链”优先筛选【系统款式】；  
* “想要显贵一点”“性价比高”等提示词 优先提升【显贵款】的权重；  
* “想要珍珠/红宝石”优先匹配【主材质 / 配石材质】；  
* “脸型匹配”不直接依赖新增商品字段，而依赖【系统属性】与内部规则表的映射关系。

## **3\. 产品范围**

### **3.1 第一阶段范围（必须实现）**

1. 支持文本对话输入；  
2. 支持图片上传；  
3. 支持预算、品类、送礼对象、风格倾向等需求识别；  
4. 支持基于货盘表的商品检索与推荐；  
5. 支持多轮追问以补全缺失条件；  
6. 支持输出商品名称、图片/二维码、价格参考、推荐理由；  
7. 支持会话记录与推荐日志记录。

### **3.2 第一阶段不做内容**

1. 不开发独立商城前端；  
2. 不新增商品字段；  
3. 不做 AR 虚拟试戴；  
4. 不做复杂会员系统；  
5. 不做自动营销推送；  
6. 不做完全自动成交闭环，保留人工承接空间。

## **4\. 核心业务流程**

### **4.1 标准流程**

1. 客户通过企业微信等官方入口发起咨询；  
2. 客户输入需求，或上传照片；  
3. 大模型先识别当前轮输入意图、抽取条件、判断是否存在条件变更；  
4. 会话状态模块合并本轮新条件，并与上一轮状态进行差异比对；  
5. 若条件不足，则由大模型生成更自然的开场白或追问语句；  
6. 若条件满足推荐要求，智能体调用商品检索能力，筛选真实商品；  
7. 对候选商品进行排序；  
8. 大模型基于本轮最终条件和候选商品生成 3–4 款推荐商品；  
9. 返回商品二维码，引导用户进入现有小程序查看或下单。

#### **4.1.1 单轮消息处理闭环（新增）**

每一轮用户消息都必须经过以下处理闭环，不允许只在“推荐阶段”才调用大模型：

1. **消息接收**：接收本轮文本、图片、会话 ID、用户 ID；  
2. **大模型语义解析**：识别本轮消息属于开场、补充条件、修改条件、追问回复、闲聊、推荐确认等哪一类；  
3. **条件抽取与归一化**：抽取预算、品类、材质、送礼对象、风格、显贵诉求、星座、生肖、生日、图片偏好等信息，并映射到标准条件字段；  
4. **状态差异判断**：判断本轮输入是新增条件、修改已有条件、否定已有条件，还是无效闲聊；  
5. **策略决策**：由大模型与规则引擎共同决定本轮应执行“开场引导 / 继续追问 / 重新检索 / 重新排序 / 直接推荐 / 解释说明”；  
6. **检索与排序**：只要关键条件发生变化，就必须重新执行商品检索与排序，不能直接复用旧推荐结果；  
7. **结果表达**：由大模型生成最终对用户可见的话术，保证语气自然、亲切、有引导感，而不是机械字段罗列。

**特别要求：**

* 当用户仅输入“你好”“在吗”“想看看珠宝”等开场语时，智能体必须先进行欢迎式开场，介绍可提供的帮助，再自然追问 1–2 个关键条件；不得直接以“你的预算是什么 / 款式是什么”这类生硬脚本开头。  
* 当用户补充“我是天蝎座”“属龙”“预算改成 5000”“想换成项链”“想要轻奢一点”时，系统必须视为新条件输入，并重新触发条件识别、规则映射、商品检索和推荐生成。  
* 当用户新输入与上一轮条件冲突时，以用户最新明确表达为准覆盖旧条件，并在必要时通过自然语言进行简短确认。

### **4.2 输入信息类型**

用户可能提供的信息包括：

* 预算；  
* 商品品类；  
* 材质偏好；  
* 送礼对象；  
* 使用场景；  
* 风格倾向；  
* 星座 / 生肖 / 生日；  
* 自拍照或参考图；  
* “显贵”“性价比高”等价值偏好。

### **4.3 推荐结果约束**

每次推荐默认输出 3–4 款商品。推荐商品必须满足：

* 商品编码真实存在；  
* 商品二维码可用，若不可用则降级展示商品图片；  
* 商品信息来自货盘表，不允许模型虚构；  
* 商品与用户需求至少存在一个明确匹配点。

## **5\. 推荐策略设计**

### **5.1 总体策略**

系统采用\*\*“需求识别 → 条件补全 → 商品过滤 → 商品排序 → 文案生成”\*\*的推荐流程。

本项目采用\*\*“大模型全链路参与 + 后端检索筛选”\*\*的策略。大模型不是只在个别节点触发，而是每一轮对话的必经环节。

**大模型主要负责：**

* 理解用户表达；  
* 提取条件；  
* 识别用户真实意图与潜台词；  
* 判断是否需要追问；  
* 判断用户是在新增条件、修改条件还是撤回条件；  
* 将自然语言偏好映射为结构化检索条件；  
* 根据检索结果组织最终输出文案；  
* 在开场、追问、解释、推荐、变更确认等所有回复类型中生成自然话术。

**检索系统主要负责：**

* 基于货盘字段筛选真实商品；  
* 返回可推荐候选商品。

### **5.1.1 大模型调用机制（新增）**

为避免系统退化为静态问答脚本，必须明确以下机制：

1. **每轮必调**：每次收到用户新消息后，必须至少执行一次大模型语义理解；若最终需要对用户输出自然语言，则该输出也必须由大模型生成或润色。  
2. **开场必调**：用户首次进入会话或仅发送寒暄语时，必须调用大模型生成欢迎语和引导语。  
3. **条件变更必调**：用户补充、修改预算、款式、材质、送礼对象、风格、星座、生肖、生日等条件时，必须再次调用大模型识别变化。  
4. **推荐前必调**：在商品列表检索完成后，必须调用大模型基于“用户条件 + 候选商品”生成推荐理由、搭配建议和引导话术。  
5. **不可只依赖固定规则模板直接回复用户**：规则引擎负责筛选，面向用户的最终表达仍需由大模型负责。

### **5.1.2 大模型与规则引擎的分工边界（新增）**

* **大模型负责理解与编排**：把“性价比高”“轻奢”“想显贵一点”“送妈妈但别太老气”“预算再往上提一点”这类自然语言转成可执行条件。  
* **规则引擎负责确定性筛选**：按预算范围、品类、材质、二维码可用性等硬条件过滤真实商品。  
* **排序模块负责显式加权**：对适合人群、显贵款、系统属性、卖点、折扣等字段进行打分。  
* **大模型负责结果表达**：把最终候选商品组织成更符合当前语境的推荐说明，并解释“为什么这几款适合您”。

### **5.1.3 条件变更重算机制（新增）**

系统必须维护“当前会话条件状态”。任一轮输入后都要判断是否发生以下变化：

* 新增条件：例如新增星座、生肖、送礼对象、风格；  
* 修改条件：例如预算从 3000 改为 5000，款式从耳饰改为项链；  
* 否定条件：例如“不想要珍珠了”“不要太成熟的”；  
* 弱化条件：例如“预算可以再灵活一点”。

一旦发生上述变化：

1. 更新会话中的结构化条件；  
2. 重新触发相应规则映射；  
3. 重新检索商品候选集；  
4. 重新排序；  
5. 重新生成推荐文案。  

**禁止出现以下情况：**

* 用户已修改预算，但系统仍沿用旧预算的推荐结果；  
* 用户新增星座/生肖后，系统输出与之前完全相同且无解释的推荐结果；  
* 用户只发送“你好”，系统直接进入生硬问参流程。

### **5.2 条件优先级**

* **一级条件（强过滤）**，优先用于直接过滤商品：  
  1. 系统款式（如项链、手链、耳饰等）  
  2. 预算（统一按批发裸价，默认按用户预算上下浮动 ±15% 召回；命中不足时可放宽至 ±20%）  
  3. 主材质 / 配石材质  
  4. 商品是否存在二维码或图片  
* **二级条件（加权排序）**，用于候选商品排序：  
  1. 适合人群  
  2. 显贵款  
  3. 系统属性  
  4. 产品优势卖点  
  5. 折扣 / 价格优势  
* **三级条件（文案增强与辅助加权）**，用于辅助推荐解释或弱加权：  
  1. 图片分析结果  
  2. 星座/生肖规则映射结果  
  3. 使用场景判断  
  4. 风格倾向判断

### **5.3 预算推荐规则**

当用户输入预算时，系统统一按\*\*【批发裸价】\*\*进行预算匹配。

* 为提升推荐结果数量与灵活性，系统默认按用户预算上下浮动 **±15%** 的范围进行商品召回。  
* 若召回结果不足 3 款，则可自动放宽至 **±20%**。  
* 若召回结果仍不足，则智能体应主动引导用户适当放宽预算或补充其他筛选条件。  
* 若召回结果过多，则系统结合适合人群、显贵款、系统属性、产品优势卖点等因素进行排序。

*示例：*

* 用户预算：3000 元  
* 第一轮召回范围：2550–3450 元（±15%）  
* 若命中不足，则放宽至：2400–3600 元（±20%）

### **5.4 多轮追问规则**

当用户输入信息不足，或命中商品范围过大时，智能体需主动追问。建议优先追问顺序如下：

1. 系统款式（想看项链、手链还是耳饰）  
2. 使用场景（自戴还是送礼）  
3. 送礼对象（送女友、妈妈、闺蜜等）  
4. 材质偏好（黄金、珍珠、红宝石等）  
5. 风格倾向（简约、温柔、大气、显贵等）

*示例：*

* 用户：“预算 3000 以内有什么推荐？”  
* AI：“3000 预算内我们有不少款式可选。为了给您推荐更合适的，想问一下您更偏项链、手链还是耳饰呢？是自己佩戴还是送人呢？”

**补充要求：**

* 追问必须建立在当前已识别条件之上，先简要承接用户输入，再提问，不可完全无上下文地重复问预算、问款式。  
* 若用户第一句只是寒暄，优先输出欢迎语、能力说明和轻量引导，而不是直接进入盘问式问答。  
* 若用户已提供预算，后续追问不应再次机械询问预算，除非预算表达不明确或存在区间冲突。  
* 若用户在已提供预算、品类或其他有效上下文后回复“随便”“都可以”“不限”“不挑”“没要求”等，应视为放开剩余约束并直接进入检索推荐，不再重复追问款式；空上下文仅表达“随便”时仍需轻量追问。
* 若用户本轮输入已经足以触发推荐，则不应继续追问，而应进入检索与推荐流程。

### **5.5 送礼场景推荐**

* **触发条件**：用户输入中包含送礼、送老婆、送女友、送妈妈、送闺蜜、送长辈等表达。  
* **处理逻辑**：  
  * 提取送礼对象；  
  * 优先匹配【适合人群】字段；  
  * 在推荐文案中加入礼赠场景表达。

### **5.6 极致性价比 / 显贵模式**

* **触发条件**：用户输入中包含“显贵”“高级感”“预算有限但想大气”“性价比高”“便宜但要好看”“轻奢”“想有质感”“不要太廉价感”“想看起来贵一点”等表达。  
* **处理逻辑**：  
  * 先由大模型识别这些自然语言是否可归入“显贵 / 轻奢 / 性价比 / 高级感”诉求；  
  * 再提升【显贵款】字段的排序权重；  
  * 结合【批发裸价】、【折扣】与【产品优势卖点】生成推荐理由；  
  * 推荐文案中可强调“粉丝可享价格”“预算友好”“显贵感强”等表达。

### **5.7 星座 / 生肖 / 生日推荐**

* **触发条件**：用户直接提供星座、生肖，或提供生日。  
* **处理逻辑**：  
  1. 若用户提供生日，系统先推算星座/生肖；  
  2. 系统查询内部“星座/生肖规则映射表”；  
  3. 将映射出的推荐材质、配石、颜色倾向转化为检索条件；  
  4. 从货盘表中匹配【主材质】、【配石材质】等字段；  
  5. 模型负责将规则解释为自然语言推荐理由。  
* **说明**：  
  * 星座/生肖推荐不直接依赖模型自由生成的内置知识；  
  * 规则内容以业务预设规则为准，便于统一口径与后续维护；  
  * 若用户在已有推荐后新增星座、生肖或生日，系统必须重新执行规则映射、候选集检索、排序和文案生成，不可直接沿用上一轮结果。

### **5.8 图片辅助推荐**

* **触发条件**：用户上传包含人脸的照片，或上传参考款式图。  
* **处理逻辑**：  
  1. 模型判断图片是否可用于分析；  
  2. 若为自拍，输出“脸部轮廓倾向 / 整体风格倾向 / 配色建议”等辅助信息；  
  3. 若为参考款式图，提取喜欢的材质、气质、设计特征；  
  4. 将分析结果映射到【系统属性】、【主材质】、【配石材质】等字段进行加权；  
  5. 最终输出“适合您当前风格”的搭配建议。  
* **说明**：  
  * 图片分析仅用于搭配建议，不作为身份识别或敏感属性判断依据；  
  * 若图片不清晰，智能体需提示用户换图或改用文本描述。

## **6\. 输出规范**

### **6.1 输出原则**

智能体输出必须满足以下原则：

* 仅推荐货盘表中真实存在的商品；  
* 每次推荐默认输出 3–4 款；  
* 推荐结果必须包含商品名称；  
* 优先输出商品二维码，二维码不可用时降级输出商品图片；  
* 推荐理由必须基于商品字段和用户输入生成，不允许无依据夸大；  
* 输出话术必须贴合当前对话阶段，开场、追问、推荐、条件变更确认应采用不同表达方式；  
* 当识别到用户新增或修改条件时，输出中应体现“已根据您刚补充/修改的信息重新筛选”这一语义。

### **6.2 标准输出内容**

每个推荐商品应尽量包含以下内容：

* 商品名称  
* 商品编码（可选展示，内部必保留）  
* 商品二维码或商品图片  
* 品类说明（读取【系统款式】）  
* 预算参考（优先读取【批发裸价】，可补充【折扣】体现粉丝权益）  
* 推荐理由与搭配建议（融合【产品优势卖点】及当前触发条件）

### **6.3 推荐文案要求**

推荐文案需满足：

1. 体现用户当前需求；  
2. 结合商品卖点；  
3. 如命中送礼、显贵、星座、图片分析等能力，应点明原因；  
4. 语气亲切、专业、具备情绪价值；  
5. 不得出现虚构、绝对化承诺或夸大性结论；  
6. 若本轮属于开场或追问阶段，应避免直接输出“正在为您推荐商品，请稍后”这类无信息量回复，除非后台确实进入异步处理且前端需要过渡提示；  
7. 若本轮属于条件变更后的重新推荐，应在文案中体现已识别到新的预算、款式、星座、生肖或风格要求。

### **6.4 不同对话阶段的话术要求（新增）**

* **开场阶段**：先欢迎，再简述可提供的帮助，最后自然追问 1–2 个最关键条件。  
* **追问阶段**：先承接已知条件，再补问缺失条件，避免重复提问。  
* **重新筛选阶段**：需明确告知用户“已按您刚补充/修改的信息重新帮您筛选”。  
* **推荐阶段**：需说明推荐命中的核心理由，如预算匹配、送礼对象匹配、显贵感、星座映射、图片风格匹配等。  
* **无结果阶段**：说明是哪些条件过严，并给出放宽建议。

### **6.5 示例模板**

为您挑选了几款更适合您当前需求的珠宝：

💎【商品名称】【商品编号】  
• 品类：\[读取系统款式\]  
• 粉丝参考价：\[读取批发裸价\]  
• 折扣信息：\[如有则读取折扣\]  
• 推荐理由：\[结合产品优势卖点 \+ 当前需求生成\]  
• 购买方式：\[展示商品二维码；若二维码不可用，则展示商品图片并引导联系客服\]

## **7\. 异常与兜底策略**

### **7.1 无匹配结果**

当当前条件下无可推荐商品时，智能体应：

* 提示当前条件过于严格；  
* 引导用户放宽预算、材质或品类条件；  
* 推荐相近预算或相近材质商品。

### **7.2 命中商品过多**

当候选商品过多时，智能体应继续追问，而不是一次性输出大量商品。

### **7.3 图片无法识别**

若图片清晰度不足、无有效人脸或主体不明确，应提示用户：

* 重新上传更清晰照片；  
* 直接描述预算、风格、品类等条件。

### **7.4 二维码缺失或失效**

若商品二维码为空、失效或无法展示：

* 降级展示【商品图片】；  
* 同时提示用户联系客服或查看相似款；  
* 记录日志以供运营补数据。

### **7.5 商品信息冲突**

若同一商品存在价格字段冲突、图片缺失或编码异常：

* 不作为优先推荐；  
* 必要时从推荐结果中排除；  
* 记录异常日志。

### **7.6 会话状态未刷新（新增）**

若用户已明确补充或修改条件，但系统仍返回旧结果或未体现新条件，则视为严重会话状态错误。系统应：

* 强制重新执行本轮条件解析；  
* 强制刷新会话状态并重新检索；  
* 在日志中记录“条件变更未生效”的异常类型；  
* 作为验收与上线前重点测试项。

## **8\. 技术架构要求**

### **8.1 核心依赖**

* **多模态大模型**：用于文本理解、图片理解、多轮对话和文案生成；  
* **商品检索服务**：用于查询 Excel 解析后的货盘数据；  
* **企业微信/官方对话入口**：作为消息接入与结果展示渠道；  
* **日志与监控服务**：用于记录推荐链路和异常情况。

### **8.2 系统模块**

建议系统拆分为以下模块：

1. **消息接入模块**：接收文本、图片、用户标识、会话 ID；  
2. **会话管理模块**：维护上下文和多轮对话状态；  
3. **大模型编排模块**：负责每轮消息的意图识别、条件提取、状态判断、回复策略决策；  
4. **条件提取模块**：提取预算、品类、材质、送礼对象、风格、显贵诉求、星座生肖等条件；  
5. **状态差异判断模块**：识别本轮是新增、修改、撤回还是延续旧条件；  
6. **图片分析模块**：处理自拍或参考图；  
7. **商品检索模块**：基于现有字段检索商品；  
8. **排序模块**：按业务规则对候选商品排序；  
9. **文案生成模块**：生成推荐理由和购买引导语；  
10. **二维码输出模块**：返回商品二维码或降级图片；  
11. **日志模块**：记录输入、输出、异常和推荐结果。

### **8.3 Prompt 基本约束**

系统 Prompt 需体现以下约束：

* 智能体角色为珠宝导购顾问；  
* 必须从货盘数据中选取真实商品；  
* 绝对禁止虚构商品编码、价格、二维码；  
* 当条件不足时必须优先追问；  
* 图片分析只可用于搭配建议；  
* 每轮收到用户新消息后，必须重新解析最新输入并与会话状态合并；  
* 当用户仅发送寒暄语时，必须先欢迎并做轻量引导，不得直接盘问式追问；  
* 当用户修改预算、款式、材质、星座、生肖、风格等条件时，必须触发重新检索与重新推荐；  
* “性价比高”“轻奢”“高级感”“显贵”“预算有限但想大气”等表达，应视为需要大模型识别的隐含偏好，而不是仅靠关键词硬匹配；  
* 最终输出需满足约定的内容结构。

### **8.4 技术栈建议**

为保证本项目能够快速落地、便于接入多模态模型、支持 Excel 数据解析及后续规则扩展。

#### **8.4.1 总体原则**

本项目的技术选型应满足以下目标：

* 便于接入大模型与多模态能力；  
* 便于处理 Excel 数据、规则映射和商品检索；  
* 便于快速开发与后续扩展；  
* 便于对接企业微信消息入口；  
* 便于记录日志、排查推荐问题与持续优化推荐规则。

#### **8.4.2 推荐技术栈**

**后端开发语言**

建议采用 Python 作为后端主要开发语言。

*选择原因：*

* Python 在大模型、多模态模型、RAG、规则引擎、数据处理等 AI 场景中生态成熟；  
* Python 处理 Excel、CSV、数据清洗和定时同步任务更高效；  
* Python 更适合快速开发业务智能体、推荐逻辑和对话服务；  
* 后续若增加图片分析、规则映射、日志分析等能力，扩展成本较低。

**后端框架**

建议使用 FastAPI 作为后端服务框架。

*选择原因：*

* 开发 API 效率高；  
* 适合构建 AI 服务与业务智能体服务；  
* 支持异步处理，便于接入模型调用和企业微信消息回调；  
* 自动生成接口文档，便于联调和测试。

**数据库**

建议使用 MySQL 或 PostgreSQL 作为结构化数据库，用于存储：

* 商品数据；  
* 商品二维码信息；  
* 会话记录；  
* 推荐日志；  
* 星座/生肖规则映射表；  
* 系统属性映射规则。

**文件与图片存储**

建议使用对象存储服务（如 OSS / COS / S3 类方案）存储：

* 商品图片；  
* 商品二维码图片；  
* 用户上传图片；  
* 导入过程中的临时文件。

**数据同步**

建议采用 Python 定时任务 或任务调度服务实现 Excel 货盘表的定时同步。

同步建议包括：

* 每日定时同步；  
* 同步后覆盖数据库中的商品信息；  
* 对商品二维码缺失、图片缺失、价格异常等情况记录日志；  
* 对失效商品或异常商品进行标记或过滤。

**AI 模型接入**

建议通过后端统一接入多模态大模型，不在渠道端直接调用模型接口。

后端负责：

* 用户文本消息接收；  
* 图片接收与分析；  
* 条件提取；  
* 商品检索；  
* 推荐排序；  
* 推荐文案生成；  
* 二维码输出。

**消息入口**

建议以 企业微信 作为当前阶段的主要消息入口。

原因包括：

* 客户信任度更高；  
* 更适合高客单价珠宝业务；  
* 便于 AI 首轮接待与人工后续承接；  
* 无需优先开发复杂前端产品。

**管理后台**

第一阶段可不单独开发复杂管理后台。

如后续需要补充运营后台，建议采用：

* 后端：复用 Python \+ FastAPI 服务

#### **8.4.3 推荐系统架构**

建议采用以下技术架构：

* **企业微信消息入口层**  
  * 接收客户文本与图片消息；  
  * 返回推荐结果。  
* **业务智能体服务层（Python \+ FastAPI）**  
  * 维护多轮对话；  
  * 提取预算、品类、送礼对象、材质等条件；  
  * 调用图片分析与商品检索能力；  
  * 生成推荐结果。  
* **商品检索与规则层**  
  * 基于 Excel 导入后的数据库进行商品筛选；  
  * 实现预算浮动召回规则；  
  * 实现显贵款加权、适合人群匹配、系统属性映射、星座/生肖规则映射等逻辑。  
* **数据存储层（MySQL / PostgreSQL）**  
  * 存储商品数据、二维码、会话记录、推荐日志、规则映射数据。  
* **对象存储层**  
  * 存储商品图片、用户上传图片、二维码资源。

#### **8.4.4 技术选型结论**

综合业务需求与开发效率，建议本项目采用以下主技术栈：

* **后端语言**：Python  
* **后端框架**：FastAPI  
* **数据库**：MySQL 或 PostgreSQL  
* **对象存储**：OSS / COS / S3 类方案  
* **消息入口**：企业微信  
* **数据同步**：Python 定时任务  
* **模型接入方式**：后端统一接入多模态模型 API

#### **8.4.5 可选备选方案**

若公司现有技术团队以其他语言为主，也可采用以下备选方案：

* **方案 A：Python \+ FastAPI（推荐）**  
  适合 AI 接入优先、开发速度优先的情况。  
* **方案 B：Java \+ Spring Boot**  
  适合已有成熟 Java 团队、强调工程规范和长期维护的情况。  
* **方案 C：Node.js \+ NestJS**  
  适合前后端统一 TypeScript 技术栈的情况。

*但从本项目的 AI 场景适配度、Excel 数据处理效率和开发效率来看，第一推荐仍为 Python \+ FastAPI。*

## **9\. 非功能性需求**

### **9.1 响应时间**

* 普通文本推荐建议控制在可接受范围内；  
* 涉及图片分析时允许更长响应时间；  
* 若响应超过5秒，渠道侧应支持“正在为您分析图片/筛选商品”等过渡提示。

### **9.2 数据同步**

* 货盘表变动后，系统需支持定时同步；  
* 同步后应尽快生效，避免推荐到失效商品；  
* 建议至少支持每日同步，后续按业务情况升级。

### **9.3 可观测性**

系统需记录以下日志：

* 用户输入内容；  
* 条件提取结果；  
* 候选商品列表；  
* 最终推荐商品；  
* 二维码是否可用；  
* 异常情况。

## **10\. 隐私与合规要求**

### **10.1 图片使用说明**

用户上传图片仅用于本次珠宝风格分析与商品推荐，不用于身份识别，不输出敏感属性结论。

### **10.2 数据存储原则**

* 用户图片与会话记录的保存周期需按公司合规要求执行；  
* 不得将用户上传图片用于模型训练；  
* 应支持按内部规范进行删除或脱敏处理。

### **10.3 输出限制**

智能体不得输出以下内容：

* 身份识别结论；  
* 涉及种族、年龄、健康等敏感判断；  
* 绝对化承诺；  
* 与商品事实不符的信息。

## **11\. 验收要点**

第一阶段验收重点包括：

1. 是否能基于现有字段完成推荐；  
2. 是否能完成多轮追问；  
3. 是否能输出真实商品与有效二维码；  
4. 是否能在图片场景下给出辅助建议；  
5. 是否具备无匹配、二维码异常、图片失败等兜底策略；
